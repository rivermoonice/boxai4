After a few sessions of asking Hermes one-off questions, you'll notice a pattern: you keep asking the same kind of thing. *"Summarise this article."* *"Format this as a table."* The point of a Skill is to stop re-asking. The point of a Skill Bundle is to stop re-asking *five things in a row*.

This is lesson four. It assumes you have Hermes installed ([L02 — Install & first run](/lessons/hermes/l02-install-and-first-run/)) and you're comfortable with slash commands ([L03 — TUI vs Desktop](/lessons/hermes/l03-daily-driver-tui-vs-desktop/)).

## The hook

Pick a workflow you do every week. **"Morning research: discover sources, pull articles, extract key points, organise them, write a brief."** Five steps. If you ask Hermes to do the whole thing in one prompt, you'll get an inconsistent answer — maybe it skips extraction, maybe it skips organisation. AI agents are *probabilistic*: the same prompt produces different results on different runs. Ron puts it like this: *"AI agents are probabilistic in nature. You cannot assume that they know all of the skills you want them to use to finish those tasks."*

Two ideas solve this:

1. **A Skill** — a markdown file that tells Hermes how to do *one specific job*. The file becomes a slash command. Type `/standup` and Hermes reads the file.
2. **A Skill Bundle** — a *group* of Skills loaded together. Type `/research` and Hermes loads all five Skills at once, in the order you specified, with an `instruction` field that sets the workflow.

That's the whole lesson. The rest is mechanics and one exercise.

## The mental model

A Skill is a command. A Bundle is a workflow.

<Picture src="/lessons/hermes/L04/skills-vs-bundles.svg" alt="Diagram: left, a single skill is one markdown file triggered by one slash command; right, a skill bundle is many skills loaded together by a single slash command, with coordination arrows from the bundle to each skill" width={900} height={480} formats={["avif", "webp"]} />

**A Skill is a markdown file in `~/.hermes/skills/`.** The filename minus `.md` is the slash command. `standup.md` becomes `/standup`. When you type it, Hermes reads the file and follows its instructions. You can write Skills by hand, or ask Hermes to write one after a workflow worked well. The second path is the magic moment: the agent notices it did something useful and saves the recipe.

**A Skill Bundle is a small YAML file in `~/.hermes/skill-bundles/`.** It names a group of Skills and an `instruction` field. The YAML's `name` becomes the slash command. Type `/research` and Hermes loads every Skill in the list and applies the `instruction` to chain them. As Ron describes it: *"skill bundles is a way to fix that by giving you a named set of skills behind the slash bundles command."*

The shape of a Bundle:

```yaml
name: research
description: General-purpose research workflow
skills:
  - web-discovery
  - article-extraction
  - topic-organisation
  - synthesis-analysis
  - report-output
instruction: |
  When invoked, produce a research report covering the topic end-to-end.
  Use the discovery Skills first, then extraction, then organisation, then
  analysis, then output. Include citations.
```

The `instruction` field is the part most people miss. It's a *"mini agents.md"* — house rules baked in once so you don't retype them. *"Always cite sources." "Use the last 7 days only." "Output as markdown with a TOC."* Specific instructions are what make Bundle output consistent across runs.

<Callout type="tip" title="Ron's co-design tip">
*"Consult with Hermes first. Ask the agent which Skills it would group for your workflow, then review the plan, then commit."* — Ron, source video `_A02brv2Csg`. Bundles you design alone are usually worse than Bundles you co-design with the agent.
</Callout>

**One honest limit to know up front.** As of June 2026, Skill Bundles are *not adaptive*. They run Skills in the order you specify. If your workflow needs different Skills based on intermediate results, split it into multiple Bundles, or write a single Skill with branching logic.

## Pick your tool

The decision tree is small. Three questions:

1. **Will I ever do this again?** If no, just type the prompt.
2. **Is this one job?** If yes, make a Skill. The third time you do a task, save it.
3. **Does it need three or more Skills together?** If yes, make a Bundle.

The point is to avoid both extremes: fifty orphan Skills you never use, and one mega-Skill that does five things badly.

| Situation | Right answer |
|---|---|
| One-off question you'll never repeat | Just type the prompt |
| Task you've done three times | Make a Skill |
| Three or more Skills that always go together | Make a Bundle |
| Multi-step workflow with branching | Multiple Bundles or one Skill with branching |
| Workflow the whole team should use | Commit the Bundle YAML to a shared repo |
| Agent just did something useful and unexpected | Let it auto-save a Skill |

<Callout type="gotcha" title="Beginner trap">
Don't over-Bundle. Three to five Bundles covers most daily-driver work. A Bundle for a one-off task is just a Skill wearing a trench coat.
</Callout>

### The day-to-day pattern

Once you have a few Bundles built, the rhythm looks like this:

- **Morning:** `/research <topic>` — research Bundle does the news scan.
- **Before standup:** `/standup` — summarises overnight activity.
- **Before code review:** `/pr-review` — pulls the diff and critiques it.
- **End of day:** `/wrap-up` — commits work and drafts tomorrow's TODO.

One Bundle per recurring workflow. Three to five Bundles covers most daily-driver use. On a team, commit the YAMLs to a shared repo — Ron's recommendation: *"commit the bundle files into your dot files or a shared repo."*

## Try it

### The exercise

Pick a recurring workflow you do at least weekly. Morning research. End-of-day wrap-up. PR review. Anything with three or more steps.

**Step 1 — Co-design with Hermes.** In the TUI, ask:

> *"Suggest which Skills we need to bundle for a [your workflow name] workflow."*

Hermes reads its own Skill catalog and proposes a grouping. Read the plan, edit it, ask again. Iterate.

**Step 2 — Create the Bundle.**

```
hermes bundles create <your-workflow-name>
```

**Step 3 — Edit the YAML.** Open `~/.hermes/skill-bundles/<your-bundle>.yaml` and tighten the `instruction` field. Be specific: *"Use only the last 7 days. Always include three citations. Output as markdown with a TOC."* Vague instructions produce inconsistent output.

**Step 4 — Reload and test.**

```
hermes bundles reload
```

Invoke your Bundle with `/<your-workflow-name>` three times on the same topic. Same output structure all three times = good `instruction`. Wild variation = needs more specificity.

**Step 5 — Share it.** Commit the YAML to a shared repo or dotfiles. That's how Bundles become a team standard.

### Success criteria

- All three test invocations produce consistent output — same structure, same sections, same level of detail.
- The Bundle loads in under a second (no expensive one-time work in any of the Skills).
- You can describe the workflow in one sentence to a teammate who has never seen it.
