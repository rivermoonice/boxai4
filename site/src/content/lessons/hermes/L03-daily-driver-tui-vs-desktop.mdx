If you've been following the academy in order, you finished [L02 — How to Install Hermes Agent for Beginners](/lessons/hermes/l02-install-and-first-run/) and ran your first sessions through the Terminal User Interface (TUI). Hermes shipped a second surface in June 2026: a native Desktop App. Jensen Huang demoed it on stage. The natural beginner question is *do I switch?*

The short answer is **no — you add**. The TUI and the Desktop App are not competitors. They are two windows into the same Hermes instance, and most power users run both at once.

## The hook

Picture your morning. You open your laptop. Three agents you left running overnight have done things. You want to know *what* before you start your day.

Open the TUI and you have to type a command — `hermes status`, scan a wall of text. It works, but it feels like reading a spreadsheet. Open the Desktop App and you get a visual dashboard: a sidebar with project folders, a status indicator showing the back-end is ready, a timeline of what each agent did while you slept. You click, you review, you decide. That's a **morning meeting**.

Now picture the afternoon. You need to ship a refactor on a 200-line function before EOD. You don't want a dashboard. You want a fast, keyboard-driven terminal where your Git branch is in the status line and you can fire commands without waiting for the previous one to finish. That's **desk work**. You open the TUI.

Different surfaces for different kinds of work. The trap beginners fall into is treating one as "the better interface" and ignoring the other.

<Callout type="tip" title="Ron's morning-meeting framing">
"It's like a way for you to be up to date with what's going on with your projects. And then, you know, when the meeting is over, you head back to your seat, and then you start working, and that's when you switch to the TUI." — Ron, source video `QSANg6VHkXI`
</Callout>

## The mental model

The single most important fact in this lesson: **the TUI and the Desktop App are two surfaces over the same Hermes back-end**. Same config, same profiles, same skill bundles, same model, same memory, same session state. You can switch mid-task without losing anything.

The differences are entirely in the *surface* — what each interface is good at showing you and asking you to do.

<Picture src="/lessons/hermes/L03/tui-vs-desktop.svg" alt="Diagram: TUI vs Desktop App side-by-side, both connected to the same Hermes agent" width={900} height={480} formats={["avif", "webp"]} />

Think of the agent as the kitchen. The TUI is the line cook's workstation — fast, knife-ready, every tool within reach. The Desktop App is the head chef's pass — slower to look at, but you see the whole board, all the dishes in progress, what's plated and what's not. Same kitchen. Different vantage points.

**The TUI is the original Hermes interface.** Keyboard-driven. Runs in any terminal. Sends raw text to the model — nothing else. Built for terminal-native developers: live Git branch in the status line, non-blocking input (you can fire the next command before the previous response finishes), thin-client mode for connecting to a remote VPS, and clean exit with zero scrollback clutter.

**The Desktop App is a native Electron + React app** that ships with v0.16 "Surface" (June 2026). It adds visual features the TUI literally cannot render: a file-browser sidebar, side-by-side previews of generated files, drag-and-drop upload, a one-click model switcher, visual toggles for skills and MCP tools, a gateway status indicator, and a settings panel for connecting to a remote back-end.

<Callout type="tip" title="Ron's orchestration call">
"The desktop app, based on my experience, is great for orchestration tasks. You know, a human in the loop, this is where you want to use the desktop app." — Ron, source video `QSANg6VHkXI`
</Callout>

**One thing you should know up front.** The Desktop App costs roughly **2× more tokens per message** than the TUI for the same work. Every message the Desktop App sends to the model includes the UI schemas, layout state, and any background file-scanning data the app maintains. The TUI sends just your text. Same model, same answer, but the Desktop App is paying for the extra context.

If you pay your own API bill — and you do, Hermes is BYOK from [L01](/lessons/hermes/l01-what-is-hermes-agent/) — this isn't a footnote. It's a real cost. The official documentation doesn't advertise it. The source video does.

<Callout type="gotcha" title="The token-cost trap">
Beginners see the Desktop App, switch to it for everything, and watch their monthly bill roughly double. That's not a bug — that's the cost of the UI schemas the Desktop App sends with every message. Pick the surface for the work, not for the prettiness.
</Callout>

## Pick your tool

The decision rule for which interface to open has three questions:

1. **Am I writing code, or managing projects?** Writing code → TUI. Managing projects → Desktop App.
2. **Do I need to *see* state (files, models, status) without typing?** Yes → Desktop App.
3. **Am I doing this from a VPS over SSH?** Yes → TUI. The Desktop App can connect to a remote back-end, but the setup is non-trivial; start with the TUI on the VPS first.

Here's the same rule in table form:

| Workload | TUI | Desktop App | Why |
|---|---|---|---|
| Coding / shipping features | Best choice | Token-heavy | TUI's lower overhead saves real money on long sessions |
| Refactoring an existing file | Best choice | Token-heavy | Same reason |
| Quick question or chat | Fine | Fine | Either works for low volume |
| Reviewing overnight agent activity | Requires CLI commands | Best choice | Desktop App's project overview is built for this |
| Switching models | Requires `hermes model` | One click | Visual model picker |
| Drag-and-drop a file | Type the path | Best choice | Desktop UX |
| VPS-hosted back-end | Thin-client mode | Settings panel | Both work; thin client is more reliable |
| MCP-heavy sessions (v0.16) | Best choice | Discoverability still rough | Source video flags this as a known Desktop weakness |
| Human-in-the-loop workflows | Cumbersome | Best choice | Visual feedback is built for this |

**Rule of thumb:** if your work is mostly *inside a terminal* — shipping code, scripting Hermes, working over SSH — open the TUI. If your work is mostly *about the terminal* — checking what ran overnight, moving tasks between agents, reviewing files — open the Desktop App. If you do both, run both. The switch is free, and the cost of running two interfaces is much less than the cost of fighting the wrong one.

<Callout type="tip" title="Ron's pro tip">
"Open the Desktop App *and* the TUI side by side. They share state. Switching between them doesn't lose your session. Most power users in 2026 run both at once." — Ron, source video `QSANg6VHkXI`
</Callout>

A few things to keep in mind:

- **You can switch mid-session freely.** Closing the TUI doesn't kill the Hermes back-end. Don't close the TUI window expecting to "come back to it" via the Desktop App — the Desktop App opens its own session view.
- **MCPs are still rough in the Desktop App as of v0.16.** Discoverability and toggling for MCP integrations work better in the TUI today.
- **Bugs are usually back-end bugs.** If a bug shows up in the Desktop App, check the TUI first to confirm — they share a back-end.

For the deeper comparison once you've tried both, see [L09 — Hermes Agent vs Claude Code vs Mavis](/lessons/hermes/l09-hermes-vs-alternatives/).

## Try it

### The exercise

You need two interfaces open at once for this one. If you haven't installed the Desktop App yet, get it from the same place you got the TUI in [L02](/lessons/hermes/l02-install-and-first-run/).

1. Open the TUI. Start a session: ask Hermes to summarise a file you have locally — anything, a notes file, a config file, a README.
2. Open the Desktop App. Connect to the same Hermes instance. Open the same session.
3. Send one message from the TUI. Note the token count or rough cost.
4. Send the same message from the Desktop App. Note the token count or rough cost.
5. Spend 30 minutes using both. Try the file browser sidebar and the model switcher in the Desktop App. Try the Git branch status line and rapid-fire commands in the TUI.
6. Switch back and forth. Notice that state — current conversation, memory, profile — follows you.

A good outcome is one sentence in your own words answering all three of these without re-reading the lesson:

> *For (a) refactoring a 200-line function, (b) reviewing what three different agents did overnight, and (c) running an MCP-heavy workflow — which interface do I open first, and why?*

If you can answer that, the lesson landed. If you can't, re-read **The mental model**.
